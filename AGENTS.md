# Nimba — Frontend Service (AI Agent Instructions)

## Project Overview

Web client for Nimba, the internal banking credit-case platform (codename Prodigy).
Next.js 16 (App Router) + TypeScript, consuming the versioned REST API from the sibling
`app/` backend. `docs/nimba-mvp-backlog.md` (in the backend repo / shared docs) is the
single source of truth; read the relevant `NIMBA-<n>` story in full before implementing.

## Full web client — NOT a PWA

There is deliberately **no service worker and no install manifest**, now or planned. Do
not add `manifest.ts`/`manifest.json`, `app/sw.js`, `next-pwa`, or any offline caching.
The DRI analyst works from a bank workstation; the offline use case that justifies a PWA
does not exist for this product.

## Layer Model (strictly one-directional dependencies)

Routing (`app/`) → Component (`components/modules/{module}/components`) →
Cache/data (`hooks/`, TanStack Query if introduced) → Service (`services/`, `ky` calls).

- `app/` — route resolution and session checks only; renders the module's top-level
  component. No business or presentation logic.
- `components/ui/` — shadcn/ui primitives only; no domain knowledge.
- `components/modules/{module}/` — one folder per business module (`identity`,
  `credit-case`, `amortization-schedule`), split into `components/`, `hooks/`,
  `services/` as actually needed.
- `components/shared/` — cross-cutting pieces (e.g. a `TradesTable` reused by generation
  and consultation).
- Omission rule (Section 1.11): a module may omit the cache layer if nothing refreshes
  in the background after initial load (e.g. the CSV preview, a one-shot round trip).
  The component layer is never omitted.

## API & Auth Conventions

- All API calls target the backend's versioned prefix (default `/api/v1`) via the
  `NEXT_PUBLIC_API_URL` / `API_BASE_URL` environment variables. Never hardcode the host
  or version.
- The backend's OpenAPI contract is the source of truth. When a backend contract
  changes, regenerate/realign types and fix the resulting TypeScript errors in the same
  PR — never let a contract drift silently.
- Authentication lives entirely in the backend (Spring Security `httpOnly` session
  cookie). The frontend only relays the cookie (`credentials: 'include'`) and exposes a
  light `useSession` hook — no NextAuth / Better Auth, no client-stored tokens.
- Validate API response shapes with Zod before use, to catch contract drift early rather
  than crashing on an `undefined` field access.

## Engineering Rules (Hard Requirements)

1. **No AI authorship trace** — no commit, PR, comment, or header may reference "AI",
   "Claude", "Copilot", "ChatGPT", "generated by", or any equivalent, and no
   `Co-authored-by` tool line. Non-negotiable.
2. **Conventional Commits** — `<type>(<scope>): <description>` with a
   `Refs: NIMBA-<number>` trailer.
3. **Branching** — one branch per story, `nimba-{number}-{slug}`, from `develop`. Never
   commit directly to `main`/`develop`. Squash-merge via PR.
4. **No hardcoded config** — environment variables for anything environment-specific.
5. **No duplicated logic** — factor shared components/schemas; colocate Zod schemas with
   their module and derive types via `z.infer`.
6. **Run before a PR** — `pnpm lint` and `pnpm build` must both pass.
